    PrizePicks Sports Model¶

    The PrizePicks Sports Model is FORGE's multi-sport player prop analysis and grading system. It combines live projections from the PrizePicks partner API, real-time player statistics from official sports data sources (NBA Stats, NHL API, VLR), and a 4-agent swarm evaluation system powered by Claude Haiku. The model currently supports NBA, NHL, MLB, WNBA, NFL, and Valorant esports.

    Summary¶

    FORGE continuously monitors player prop lines from the PrizePicks partner API and identifies potential edges by comparing: - PrizePicks line vs. player's recent performance (L10 average, median, weighted projection) - High-confidence stat categories (e.g., Points, Rebounds, Assists for NBA; Kills for Valorant) - Variance analysis (standard deviation, hit rates, trend detection)

    When an edge is found, the model uses a 4-agent swarm evaluator that assesses the pick from four independent perspectives (Stats, Matchup, Trends, Contrarian), then a Judge synthesizes the opinions into a final verdict with confidence score. Picks are graded automatically as game results become available.

    Key Metrics (as of Mar 2026): - Overall accuracy: 70.2% across all sports - Under picks: 85.2% accuracy (slight model advantage) - Over picks: ~60% accuracy - 57+ predictions graded in the sports_predictions table - Cost: FREE (uses only free APIs)

    Sports Props Module¶

    Data Sources¶

    • PrizePicks Partner API (https://partner-api.prizepicks.com)
    • Fetches current projections and lines for all supported sports
    • Uses league IDs: NBA=7, NHL=9, MLB=2, WNBA=3, NFL=1, Valorant=159

    • No authentication required

    • Stats APIs (sport-specific)

    • NBA: nba_api Python package (stats.nba.com)
    • NHL: api-web.nhle.com official REST API
    • Valorant: vlrdevapi package (scrapes VLR.gg)

    Line Selection Strategy¶

    Standard Line Selection: PrizePicks posts the original line first, then adds variant lines (demon/goblin alternates) with slightly different odds at later timestamps.

    The model uses standard_only=True to isolate the canonical line: 1. Group all projections by (player, stat_type) tuple 2. Sort each group by board_time (earliest = original) 3. Keep only the first (standard) line per player+stat

    This filters ~250 raw lines down to ~100-150 actionable standard lines per sport query.

    High-Confidence Stats: Only models stats with predictable patterns: - NBA: Points, Rebounds, Assists, Pts+Rebs+Asts, 3-Pt Made, Blks+Stls, Turnovers, Fantasy Score - NHL: Points, Shots On Goal, Goals, Assists, Blocked Shots, Saves, Goalies - Saves, Fantasy Score - MLB: Total Bases, Hits+Runs+RBIs, Pitcher Strikeouts, Hits Allowed, Walks Allowed, Fantasy Score

    Median vs. Mean¶

    The model calculates both L10 average and L10 median because: - Averages can be skewed by outlier games (player with 45 points in a blowout) - Medians better reflect consistent baseline performance - High standard deviation signals low predictability

    Weighted Projection¶

    Recent games weighted heavier than older games: 

    Weighted Avg = (G1×w1 + G2×w2 + ... + G10×w10) / sum(weights)
    Weights decay: most recent game = 1.0, oldest = ~0.3

    Hit Rate Calculation¶

    For each stat, tracks: - Hit Rate Over: % of games player exceeded the line - Hit Rate Under: % of games player fell short - Edge %: (player_avg - line) / line × 100

    Example: Player averages 25.3 pts, line is 24.5, edge = +3.3% OVER

    Swarm Evaluator¶

    Architecture¶

    The swarm is a 5-model consensus system that evaluates each pick in parallel:

           PICK (player, stat, line, direction)
            |
         [PARALLEL AGENTS - 4 Haiku calls]
            |
    +-------+-------+-------+--------+
    |       |       |       |        |
   Stats  Matchup Trends Contrarian [Judge]
  Analyst Analyst Analyst  Analyst   |
    |       |       |       |        |
    +-------+-------+-------+--------+
            |
       FINAL VERDICT
    (OVER/UNDER/SKIP, confidence 0-100)

    Agent Roles¶

    1. Stats Analyst — Pure quantitative analysis - Compares L10 average vs. line gap - Assesses hit rate consistency - Flags high variance as unpredictable - Weights: median, standard deviation, sample size - Verdict: OVER, UNDER, or SKIP

    2. Matchup Analyst — Context from opponent + situational factors - Research opponent's defensive rank for this stat - Account for pace of play (fast teams inflate stats) - Consider position-specific matchups - Flag special circumstances (rival, playoff implications) - Verdict: OVER, UNDER, or SKIP

    3. Trends Analyst — Momentum and situational patterns - Hot streak detection: recent games > L10 avg → trending up - Rest patterns: back-to-back games = ~3-5% stat decline - Home vs. away splits - Fatigue and motivation signals (time of season) - Verdict: OVER, UNDER, or SKIP

    4. Contrarian Analyst — Devil's advocate role - Identifies "trap lines" (lines that look too easy) - Regression to the mean: hot/cold streaks revert - Sample size bias: 10 games is small, anything can happen - Blowout risk: player benched if team is up/down 30+ - Questions if public consensus is wrong - Verdict: AGREE (with the consensus pick), FADE (oppose it), or SKIP

    Judge¶

    Synthesizes all four opinions using weighted consensus: - Stats analyst: Highest weight (40%) - Matchup analyst: Medium-high weight (30%) - Trends analyst: Medium weight (20%) - Contrarian analyst: Medium weight (20%)

    If 3+ agents agree on direction → lean toward that consensus. If contrarian raises a strong concern → reduce confidence.

    Final Output: {verdict: OVER|UNDER|SKIP, confidence: 0-100, reason: "..."}

    Cost¶

    Each pick evaluation costs ~$0.02 (4 Haiku calls + 1 judge call at $0.80/1M input tokens, ~25 tokens per call).

    NBA Pipeline¶

    Data Retrieval¶

    Uses the nba_api Python package with official NBA Stats API: - Endpoint: https://stats.nba.com/stats - Requires: Player ID (from nba_api's static player registry) - Game logs: 2025-26 season (current)

    Player Matching¶

    Normalizes player names to handle accents and formatting: 

    normalized = unicodedata.normalize("NFD", "Luka Dončić".lower())
normalized = "".join(c for c in normalized if unicodedata.category(c) != "Mn")
# Result: "luka doncic"

    Then searches by: 1. Last name first (most likely to be unique) 2. Fallback: First name 3. Filters: Active players only (ignores retired/inactive) 4. Scoring: Best name match from candidates

    Statistics Tracked¶

    Per-game: PTS, REB, AST, STL, BLK, TOV, FG3M, FGA, FGM, FTA, FTM, MIN

    Combo stats auto-calculated: - PTS+REB+AST (fantasy standard) - PTS+AST, PTS+REB, REB+AST (popular flex plays) - STL+BLK (defensive production)

    Recency Weighting¶

    Recent games weighted more heavily: 

    weights = [1.0, 0.95, 0.90, 0.85, 0.80, 0.75, 0.70, 0.65, 0.60, 0.50]
# Game 1 (oldest) → Game 10 (most recent)

    Timeout Handling¶

    nba_api occasionally times out (official API is rate-limited). The module: 1. Retries up to 3 times with exponential backoff 2. Uses 45-second timeout per request 3. Runs in executor (non-blocking async)

    NHL Pipeline¶

    Data Source¶

    api-web.nhle.com — Official NHL Stats API - Free, no authentication, no rate limits - Supports both skaters and goalies - Per-game splits and aggregates

    Goalie-Specific Stats¶

    NHL props frequently feature goalie stats (especially Saves and Saves Against). The module tracks: - Saves (official stat) - Goalies - Saves (PrizePicks naming convention) - Goals Against, Win/Loss

    Standardization¶

    NHL stats names vary between PrizePicks and the official API. The module normalizes: - Points (Skater) → sum of Goals + Assists - Shots On Goal (Skater) - Blocked Shots (Skater) - Saves (Goalie)

    VLR / Valorant Esports¶

    Data Sources¶

    • VLR.gg Match Data — via vlrdevapi Python package
    • Player search and profile lookups
    • Agent-specific stats (kills, deaths, ACS, rating, KAST per agent)
    • Match schedules (completed and upcoming)

    • Per-map kill counts via HTML scraping

    • PrizePicks Valorant Lines — Partner API with league_id=159

    Player Lookup¶

    search_player(name: str) returns player profile: 

    {
  "player_id": 12345,
  "ign": "starxo",
  "country": "Denmark",
  "team": "Gentle Mates"
}

    Caches lookups to avoid repeated searches.

    Player Averages¶

    get_player_averages(player_id, timespan="90d") calculates: - kills_per_round (KPR) - kills_per_map - avg_acs (Average Combat Score, weighted by rounds played) - avg_rating (VLR performance rating) - avg_kast (Kill, Assist, Survival, Traded percentage) - avg_kd (Kill-to-Death ratio) - Agent breakdown: Usage count and stats per agent (Jett, Sage, Omen, etc.)

    30-minute cache to avoid re-scraping VLR within short intervals.

    Per-Map Kill History¶

    get_player_match_kills(player_id, limit=10) scrapes VLR match pages for: 

    {
  "match_id": 12345,
  "map_name": "Abyss",
  "kills": 18,
  "deaths": 5,
  "assists": 2,
  "rating": 1.85,
  "acs": 250,
  "date": "2026-03-28",
  "opponent": "NRG"
}

    Parses HTML tables from VLR match detail pages (.vm-stats-game selector).

    CPU Optimization¶

    VLR scraping can burn 95% CPU if not filtered. The module: - Filters to Valorant-only picks before scraping - Caches results aggressively (30-min TTL on both player averages and match kills) - Limits match history to 10 most recent - Runs BeautifulSoup parsing in a controlled manner

    See [[feedback_vlr_cpu_burn]] for performance fix applied Mar 2026.

    Auto-Grading¶

    Prediction Storage¶

    All picks are stored in sports_predictions table: 

    id | player | team | sport | stat_type | direction | line | 
projected | graded_result | verdict | swarm_confidence | 
created_at | game_date | game_id | resolved_at

    Grading Process¶

    After game resolution: 1. Query final player stats for the day 2. Match each pick to its game result 3. Determine if player exceeded line (OVER) or fell short (UNDER) 4. Update graded_result and resolved_at

    Automatic grading runs via: Cron job in cogs/prizepicks_cog.py (daily at 2 AM PST)

    Accuracy Calculation¶

    get_stats() returns win rate by category: 

    {
  "overall": {
    "win_rate": 0.702,      # 70.2% accuracy
    "total": 57,
    "wins": 40,
    "losses": 17
  },
  "by_direction": {
    "OVER": {"wins": 21, "losses": 14, "rate": 0.60},
    "UNDER": {"wins": 19, "losses": 4, "rate": 0.852}
  },
  "by_sport": {
    "NBA": {"wins": 25, "losses": 10, "rate": 0.714},
    "Valorant": {"wins": 5, "losses": 3, "rate": 0.625}
  }
}

    Edge Case Handling¶

    • Cancelled games: Picks marked as "no_action"
    • Injured players: Flagged pre-game if scrape detects IL status
    • Line movement: Final line at game start used for grading (if available)
    • Prop mismatches: Pick for "Maps 1-2" graded only on those maps (Valorant)

    Key Files¶

    File Purpose
    modules/sports_props.py Core props fetching, line selection, stat aggregation (NBA, NHL, MLB)
    modules/sports_swarm.py 4-agent swarm evaluator + judge synthesis (all sports)
    modules/sports.py Live scores, standings, schedules via ESPN API (secondary system)
    modules/vlr.py Valorant player lookup, match history, auto-grading against VLR results
    modules/predictions.py Prediction markets (Polymarket, Kalshi) — separate from sports props
    cogs/prizepicks_cog.py Discord commands: slip tracking, screenshot parsing, pick history
    cogs/sports_cog.py Discord commands: live scores, standings, schedules (ESPN)
    cogs/predictions_cog.py Discord commands: prediction market trends, category views

    Integration Points¶

    Discord Commands¶

    • /picks report → Daily picks report (top edges, confidence, results)
    • /picks slip → Manual slip entry or parse screenshot
    • /picks history → Win/loss history and accuracy stats
    • /vlr <player> → Valorant player profile + recent stats
    • /scores <league> → Live scores, standings
    • /predictions trending → Trending prediction markets

    Cron Jobs¶

    • 2 AM PST: Auto-grade daily predictions against finalized game results
    • 3 PM PST: Fetch tomorrow's PrizePicks lines for scheduled games
    • Every 4 hours: Update Valorant match schedule (live picks)

    Database Tables¶

    • sports_predictions — Graded picks (57+ rows)
    • vlr_match_results — Cached VLR match data
    • prizepicks_slips — User bet tracking (in bets_cog)

    Model Tuning & Future Work¶

    Current Limitations¶

    • No injury detection (relies on manual flagging)
    • Matchup analyst lacks real-time opponent defensive rankings (uses team reputation)
    • Trends analyst uses only last 10 games (could expand to season-long patterns)
    • Contrarian analyst is rule-based (could use historical trap line detection)

    Improvements (Roadmap)¶

    • Integrate official injury reports (ESPN or official league APIs)
    • Opponent defensive ranking (by sport: NBA efficiency ratings, NHL Corsi, etc.)
    • Rust-based VLR scraper to replace BeautifulSoup (performance)
    • Machine learning: train model on historical graded picks to weight agents dynamically
    • Under advantage: Model already shows 85.2% accuracy on unders — could weight unders higher
