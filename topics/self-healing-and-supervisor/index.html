    Self-Healing and Supervisor Systems

    11 min read · Last updated April 04, 2026 · 2,162 words

    Last Updated: 2026-04-04
    Category: Architecture, Resilience, Operations

    Overview

    FORGE implements a three-layer autonomous defense system to achieve "Unkillable FORGE" — a Discord bot that can detect and recover from failures without human intervention. This system bridges the gap between traditional error handling and true autonomy: FORGE detects problems and fixes them independently, rather than waiting for Claude Code to intervene.

    The defense comprises:

    1. Preflight Defense (before crash): Import validation catches broken edits before restart
    2. Supervisor Process Management (during runtime): External watchdog manages bot.py lifecycle
    3. Self-Healing Engine (after detection): Autonomous cog recovery, circuit breaker adaptation, and escalation

    Layer 1: Preflight Defense

    File: forge_supervisor.py (lines 1-100+)
    Trigger: On FORGE crash (non-zero exit code)
    Cost: ~2 seconds

    Purpose

    Before restarting bot.py, the supervisor runs import checks on recently modified files. If any file fails to import, it's automatically reverted to the last known-good version.

    Implementation

    # Pseudo-logic from forge_supervisor.py
1. bot.py exits (crash detected)
2. Supervisor scans MODIFIED_FILE_WINDOW (600s = 10 min)
3. For each recently modified Python file:
   - Import file via Python interpreter
   - If ImportError → revert file from git
   - Log reversion to supervisor.log
4. Restart bot.py

    Configuration

    Parameter Value Purpose
    MAX_CONSECUTIVE_CRASHES 3 Halt restarts after N fast crashes
    FAST_CRASH_THRESHOLD 30s Crash within 30s = "fast crash"
    MODIFIED_FILE_WINDOW 600s Look back 10 minutes for changed files
    CRASH_COOLDOWN 300s Wait 5 min before retry (after halt)

    Key Files

    • State file: data/supervisor_state.json — JSON record of last crash, reverted files, state transitions
    • PID file: data/supervisor_pid.json — Supervisor process ID (for external monitoring)
    • Log: logs/supervisor.log — All supervisor actions, formatted with timestamps

    Example Scenario

    10:15:03 [FORGE.supervisor] Starting bot.py (PID 8492)
10:15:35 [FORGE.supervisor] bot.py crashed with code 1
10:15:35 [FORGE.supervisor] Scanning modified files (last 600s)
10:15:35 [FORGE.supervisor] Found: cogs/my_cog.py (modified 2min ago)
10:15:36 [FORGE.supervisor] Preflight check: import cogs.my_cog → ImportError
10:15:36 [FORGE.supervisor] Reverting cogs/my_cog.py to last git commit
10:15:37 [FORGE.supervisor] Starting bot.py (attempt 1/3)

    Limitations

    • Only catches import-time errors (syntax, missing dependencies, circular imports)
    • Does NOT catch runtime errors (those are handled by Layer 3)
    • Cannot revert files if git is unavailable or repo is corrupted

    Layer 2: Supervisor Process Management

    File: forge_supervisor.py (lines 40-300+)
    Lifecycle: Continuous during FORGE runtime

    Core Responsibilities

    1. Subprocess Management — Launches bot.py with proper I/O handling
    2. Heartbeat Monitoring — Checks data/heartbeat.json written by watchdog every 10 minutes
    3. Crash Loop Protection — Halts after MAX_CONSECUTIVE_CRASHES fast crashes
    4. Recorder Co-Management — Also launches/monitors forge_recorder.py (standalone voice recorder)
    5. State Tracking — Writes to supervisor_state.json for visibility

    Heartbeat System

    The supervisor reads a heartbeat file written by the watchdog module (every 10 minutes during active hours).

    {
  "timestamp": "2026-04-04T15:30:00Z",
  "uptime_seconds": 3600,
  "memory_mb": 425.3,
  "event_loop_lag": 0.002,
  "last_message": "2026-04-04T15:29:45Z"
}

    If the heartbeat is stale for > 20 minutes (HEARTBEAT_STALE_THRESHOLD), the supervisor considers FORGE hung and may restart it.

    State Transitions

    graph TD
    A["HEALTHY"] -->|crash detected| B["CRASHED<br/>Preflight check<br/>Revert files"]
    B -->|fast restart succeeds| A
    B -->|fast restart fails| C["CRASHED again<br/>Repeat until<br/>MAX_CRASHES"]
    C -->|MAX_CRASHES reached| D["HALTED"]
    D -->|Wait CRASH_COOLDOWN 300s| E["Reset counters"]
    E --> A

    Recorder Co-Management

    The supervisor also manages forge_recorder.py, a standalone process that: - Captures FORGE's output in a ring buffer - Survives bot crashes (separate process) - Written to in real-time by FORGE - Readable for diagnostic purposes

    Launched alongside bot.py, managed with same lifecycle.

    Layer 3: Self-Healing Engine

    Files: - core/self_healing.py — Main SelfHealer class - core/resilience.py — Circuit breaker pattern - core/error_recovery.py — Discord error recovery buttons - modules/forge_watchdog.py — Silent failure detection - modules/forge_monitor.py — Centralized error logging - modules/circuit_breaker.py — DB-backed circuit breaker state - core/forge_loop.py — Event bus wiring

    Trigger: Autonomous, scheduled + event-driven
    Scope: Self-correcting without human input

    Three Sub-Components

    3A. Cog Auto-Recovery

    When a cog (module) crashes or becomes unresponsive:

    # From self_healing.py :: CogAutoRecovery
1. Watchdog detects cog not responding to events
2. Triggers self_healer.recover_cog(cog_name)
3. Unloads broken cog with await bot.unload_extension()
4. Waits 2 seconds (resource cleanup)
5. Reloads cog with await bot.load_extension()
6. Posts recovery message to #📡-monitor

    Status tracking: Cog-level recovery tracked in database, history in watchdog_cog_recovery table.

    3B. Failure Adaptation (Circuit Breaker)

    File: core/resilience.py
    Pattern: Three-state finite state machine

    The circuit breaker wraps external API calls (Anthropic, GitHub, Apify, Tavily, Exa) with failure tracking:

    graph TD
    A["CLOSED<br/>Normal"] -->|N consecutive failures| B["OPEN<br/>Rejecting calls"]
    B -->|timeout elapsed| C["HALF_OPEN<br/>Testing recovery"]
    C -->|M successes| A
    C -->|any failure| B

    Configuration (per-service, in CircuitConfig):

    Parameter Default Meaning
    failure_threshold 5 Open after N failures
    success_threshold 2 Close after N successes in HALF_OPEN
    reset_timeout 60s Wait before trying HALF_OPEN
    half_open_max_calls 3 Max concurrent test calls

    Database-backed via modules/circuit_breaker.py:

    CREATE TABLE circuit_breakers (
  service TEXT PRIMARY KEY,
  state TEXT,        -- 'closed', 'open', 'half_open'
  failure_count INT,
  opened_at TEXT,    -- ISO8601
  cooldown_seconds INT
)

    Automatic Adaptation: The self-healer periodically reviews failure patterns and adjusts thresholds. High-failure services get stricter thresholds; recovering services get relaxed thresholds.

    3C. Friction Escalation

    File: core/self_healing.py :: FrictionEscalator

    Detects recurring error patterns and promotes them to enforced preferences:

    # Escalation workflow
1. Every 6 hours, scan friction.jsonl (error history)
2. Count patterns: "API timeout on enrichment" (3x in past week)
3. If pattern exceeds threshold:
   a. Create preference rule: "Retry enrichment API max 2 times, then skip"
   b. Write to preference_rules table
   c. Reload classifier to bake in rule
   d. Post escalation to #📡-monitor: "🔥 Escalated: Enrichment API retry limit"

    Benefits: - Learns from repeated failures - Prevents cascading failures by making recovery automatic - Reduces human intervention (no need to re-architect after pattern emerges)

    3D. Health Pulse

    Schedule: Every 15 minutes
    Scope: Periodic self-check

    Runs comprehensive health checks: - Event loop lag (should be < 0.01s) - Memory usage (alert if > 80% of baseline) - Database connections (all open?) - Scheduled tasks (all running on time?) - Discord connectivity (guild members visible?) - Last message timestamp (recent or stale?)

    Posts summary to #📡-monitor with color-coded status: - 🟢 All healthy - 🟡 Warnings (e.g., lag > 0.005s but < 0.02s) - 🔴 Critical (e.g., event loop lag > 0.05s)

    Integrated Monitoring

    Watchdog Module

    File: modules/forge_watchdog.py

    Goes beyond crash monitoring — detects silent failures:

    • Scheduled tasks not firing (briefing, email, meeting prep, autopilot)
    • Commands going unanswered
    • Stale data (weather, emails, last message timestamp)
    • API degradation (slow responses, auth failures)
    • Resource issues (memory growth, event loop lag)

    Schedule: Every 10 minutes during active hours (7am-11pm)
    Cost: $0 (pure Python introspection, no API calls)
    Alert States:

    Severity Color Meaning
    🟢 Recovery Green Was broken, now working (auto-clears after 2 consecutive passes)
    🟡 Warning Yellow Degraded but functional
    🔴 Critical Red Something is broken

    Alert state machine prevents spam: once alerted, won't re-alert same issue for 30 minutes (cooldown).

    Monitor Module

    File: modules/forge_monitor.py

    Hooks into Python's logging system to catch all ERROR+ level events:

    1. DiscordErrorHandler — Logging handler that captures logger.error() and logger.critical()
    2. Deduplication — Same error suppressed for 5 minutes (prevents spam)
    3. Rate Limiting — Queues Discord posts, processes sequentially
    4. Module Tracking — Counts errors per module, alerts if spike detected
    5. Startup Health Check — Posts boot summary to #📡-monitor on bot ready

    Every error becomes a Discord embed in #📡-monitor:

    🔴 ERROR: enrichment_api
Module: FORGE.modules.enrichment
Func: enrich_candidate_sync
Message: "API timeout after 30s"
Time: 2026-04-04 15:42:19 UTC

    Process Monitor

    File: modules/process_monitor.py

    Wires into the process registry singleton and posts lifecycle events to #📡-monitor:

    • ▶️ Started — Background task launched
    • ✅ Completed — Task finished successfully (includes duration)
    • ❌ Failed — Task crashed (includes error)
    • 🛑 Stopped — Task manually stopped

    Example embed: 

    ▶️ Started: enrichment_batch
Channel: #⚡-forge
Source: scheduler (hourly job)
Expected: 15 mins

    Independence Loop (Event Bus Wiring)

    File: core/forge_loop.py

    Connects all feedback loops via event bus subscriptions (fire-and-forget):

    Learn Loop

    graph TD
    A["Classifier corrected"] --> B["Update WAL"]
    B --> C["Pattern detection"]
    C --> D["Dynamic few-shot"]
    D --> E["Better classifier"]

    When Zack corrects the classifier, the correction is logged to the Write-Ahead Log (WAL), then used to improve the classifier's few-shot examples.

    Heal Loop

    graph TD
    A["API failure detected"] --> B["Circuit breaker trips"]
    B --> C["Self-healer notified"]
    C --> D["Threshold adaptation"]
    D --> E["Monitor alert posted"]

    Awareness Loop

    graph TD
    A["Unknown request"] --> B["Gap recorded"]
    B --> C["Brain analyzes"]
    C --> D["Suggests exact command"]

    When the classifier returns None (no match), the user's message is recorded as a capability gap, and the brain (Sonnet) later suggests which command to add or which classifier rule to fix.

    Error Recovery with User Choice

    File: core/error_recovery.py

    When an error occurs during a user-facing operation (enrichment, scraping, etc.), the bot:

    1. Detects error type via regex pattern matching
    2. Suggests recovery actions with Discord buttons
    3. Waits for user choice (retry, skip, switch provider, check logs)
    4. Acts on choice and continues operation

    Example:

    ⚠️ Enrichment API timeout after 30s
[🔄 Retry] [⏳ Wait 60s] [⏭️ Skip]

    Recovery patterns recognized:

    • Timeouts → Retry / Skip
    • Rate limits → Wait + Retry / Skip
    • Connection errors → Retry / Skip
    • Auth errors → Check Logs / Skip
    • Database errors → Retry / Check Logs
    • Circuit breaker open → Wait + Retry / Skip

    Operational Guidelines

    When to Trust Auto-Recovery

    • Simple cog reloads (network glitch, timeout)
    • API circuit breaker trips (service degraded)
    • Friction escalation (pattern-based optimization)
    • Watchdog alerts (informational, no action taken)

    When to Intervene Manually

    • Multiple consecutive crashes (supervisor halts after 3)
    • Heartbeat stale for 20+ minutes (possible infinite loop)
    • High error rate spike (> 10 errors/hour in single module)
    • Database corruption or git repo issues

    # Friction escalation every 6 hours
