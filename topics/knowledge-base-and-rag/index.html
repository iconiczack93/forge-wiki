    Knowledge Base & RAG System¶

    FORGE's knowledge base implements Retrieval Augmented Generation (RAG) to ground Claude's responses in real documents. The system combines three retrieval mechanisms: ChromaDB vector search for semantic matching, Knowledge Graph for entity relationships, and Knowledge Router for query-aware retrieval strategy selection.

    Overview¶

    The knowledge base system consists of:

    1. Vector Store (ChromaDB) — semantic document search
    2. Knowledge Graph — entity and relationship extraction
    3. Knowledge Router — intelligent query routing
    4. Memory Systems — user facts, reflections, conversation history
    5. SQLite Database — persistent storage for all features

    Queries against the knowledge base are automatically routed to the optimal retrieval mechanism based on query intent, then results are injected into Claude's system prompt.

    Architecture Diagram¶

    User Query
    ↓
[Knowledge Router] — Detect query type (entity / conceptual / hybrid)
    ↓
    ├─→ "Who is..." → [Knowledge Graph] (entity-first, + light vector)
    ├─→ "How to..." → [ChromaDB] (conceptual, vector-only)
    └─→ Default → [Graph + ChromaDB] (parallel, hybrid)
    ↓
[Merge Results] → System Prompt Context
    ↓
Claude Response

    Vector Store: ChromaDB¶

    Location: data/chroma_db/
    Module: modules/knowledge_base.py

    Features¶

    • Local, GPU-free: Uses ONNX embeddings (no GPU required)
    • Lazy-load: Defers initialization to first use, avoiding 30s boot spike
    • Auto-ingest: Watches knowledge/, jds/, and vault/ directories
    • File tracking: SQLite table kb_documents tracks ingestion state

    Supported File Types¶

    • PDF, DOCX, TXT, MD, CSV, JSON
    • Job descriptions (from jds/ directory)
    • Vault notes (from vault/daily, vault/notes, vault/meetings, etc.)

    Document Chunking¶

    CHUNK_SIZE = 500          # tokens (~2000 chars)
CHUNK_OVERLAP = 50        # tokens (~200 chars)
CHARS_PER_TOKEN = 4       # rough estimate

    Text is split into overlapping chunks to preserve context boundaries. Overlap prevents losing information at chunk edges.

    Semantic Search¶

    Score Threshold: 0.3 (cosine similarity)

    Results with scores above 0.3 are returned. The threshold balances precision (fewer false positives) with recall (finding relevant content).

    async def search(query: str, top_k: int = 5) -> list[dict]:
    """Search ChromaDB and return top K results with scores ≥ 0.3"""
    results = _collection.query(
        query_texts=[query],
        n_results=top_k * 2  # Fetch 2x, filter by threshold
    )
    # Filter: score >= 0.3
    filtered = [r for r in results if r['score'] >= 0.3]
    return filtered[:top_k]

    Lazy Loading¶

    ChromaDB is not loaded on bot startup. Instead, it's initialized on first use (_ensure_chromadb()), avoiding: - 30-second CPU spike during boot - Unnecessary loading if KB is unused - Complex async state management at startup

    When initialized, the system ingests all tracked files (JDs, knowledge/, vault/) in parallel.

    Knowledge Graph¶

    Location: modules/knowledge_graph.py

    Purpose¶

    Extracts and stores entities (people, organizations, projects, technologies) and relationships between them. Enables structured fact retrieval and entity-focused queries.

    Entity Types¶

    Relationships¶

    VALID_PREDICATES = {

    Database Tables¶

    Table Purpose
    entities Entity definitions (name, type, canonical form)
    relationships Links between entities (subject, predicate, object)
    entity_access Tracks which entities are most queried

    Entity Canonicalization¶

    Entity names are normalized to prevent duplicates: - Lowercase - Strip whitespace - Replace - with _ - Remove '

    Example: "John Smith" → "john smith", "John-Smith" → "john smith" (same entity)

    Caching¶

    Entity profiles are cached with a 10-minute TTL: 

    _entity_cache: dict[str, tuple[float, object]] = {}
_CACHE_TTL = 600  # seconds

    Prevents repeated database queries for frequently accessed entities.

    Knowledge Router¶

    Location: modules/knowledge_router.py

    Query Classification¶

    Routes queries to the optimal retrieval strategy:

    Entity Queries → Knowledge Graph + Light Vector¶

    Triggers on entity-focused prefixes: - "Who is...", "Where is...", "What is..." - "Tell me about...", "Search for...", "Info on..."

    Behavior: Fetch from knowledge graph first, supplement with 2 vector results.

    Conceptual Queries → Vector Search Only¶

    Triggers on explanatory prefixes and keywords: - "How to...", "Why does...", "Explain..." - "Compare...", "What are the differences...", "Best practice"

    Behavior: Use ChromaDB only (no graph lookup).

    Mixed/Default → Hybrid (Parallel)¶

    Triggers when no specific signals match.

    Behavior: Query both graph and vector in parallel, merge results.

    Routing Logic¶

    def route_query(query: str) -> str:
    """Returns: "entity", "conceptual", or "hybrid" """

    # 1. Check prefix matches (highest confidence)
    if query.lower().startswith(("who ", "where ", ...)):
        return "entity"

    if query.lower().startswith(("how ", "why ", ...)):
        return "conceptual"

    # 2. Check keywords anywhere in query
    if re.search(r"\b(how does|explain|trade-?offs?)\b", query):
        return "conceptual"

    # 3. Check entity patterns (proper nouns, @mentions)
    if re.search(r"[A-Z][a-z]+(?:\s+[A-Z][a-z]+)+", query):
        return "entity"

    return "hybrid"

    Memory Systems¶

    FORGE implements four complementary memory layers, each with distinct semantics and uses.

    1. Conversation Memory (Short-Term)¶

    Table: conversation_messages
    Module: modules/memory.py

    Stores every message in a channel. When retrieving context, the system fetches: 1. Summaries (if older than context window allows) 2. Recent N messages (most recent first) 3. Current message

    Use: Building multi-turn conversation context for Claude.

    2. Forge Memory (Semantic, Category-Driven)¶

    Table: None (file-based)
    File: ~/.claude/projects/C--Users-User-FORGE/memory/MEMORY.md
    Module: modules/forge_memory.py

    A semantic, hierarchical knowledge system organized into categories:

    Category Priority Auto-Decayed?
    identity 1 No (protected)
    family 2 No (protected)
    preference 3 Yes
    project 4 Yes
    decision 5 Yes
    person 6 Yes
    learning 7 Yes
    history 8 Yes

    Use: Long-term semantic knowledge about Zack, projects, decisions, and learnings. Injected into system prompt in priority order.

    Immutable Core (WORM_PERSONA): A write-once block describing FORGE's identity and capabilities, never updated.

    3. Fact Memory (Per-User, Extracted)¶

    Table: user_facts
    Module: modules/fact_memory.py

    Automatically extracts structured facts about users via Claude Haiku after each conversation.

    Data Model: 

    @dataclass(frozen=True)
class UserFact:
    id: str                    # UUID
    user_id: str               # Discord user ID
    content: str               # "User prefers dark mode"
    category: str              # preference|knowledge|context|behavior|goal
    confidence: float          # 0.0-1.0
    created_at: str            # ISO 8601
    source: str                # "discord_msg_123"

    Extraction Rules: - Confidence ≥ 0.9: explicit facts ("I live in LA") - Confidence 0.7-0.9: strongly implied facts - Confidence < 0.7: skipped

    Deduplication: UNIQUE(user_id, content) prevents duplicate facts.

    Use: Personalized responses tailored to individual user preferences, knowledge, and goals.

    4. Reflection Store (Therapy-Like Introspection)¶

    Tables: reflection_entries, reflection_sessions, therapist_memory
    Module: modules/reflection_store.py

    Structured reflection data for introspection and pattern analysis.

    Tables:

    Table Purpose Key Fields
    reflection_entries Individual Q&A entries date, session, question, answer, category
    reflection_sessions Session grouping date, status (pending/completed), questions_json, answers_json, therapist_response
    therapist_memory Extracted patterns date, note, category (observation/pattern/insight)

    Queries: - Find all reflection entries by date or category - Retrieve session progress (pending answers, completed sessions) - Extract patterns from therapist notes

    Use: Self-reflection, pattern recognition, and personal growth tracking.

    Both caches auto-prune when: - Cache exceeds size limit (50 for KB, 200 for entities) - Entry TTL expires

    Design Decisions¶

    Why Multiple Memory Systems?¶

    • Conversation Memory: Short-term, turn-by-turn context (can be summarized)
    • Forge Memory: Semantic, long-term, user-level knowledge (Zack-specific facts)
    • Fact Memory: Automatically extracted, per-user behavioral insights
    • Reflection Store: Introspection & pattern analysis

    Each addresses a distinct use case and retrieval pattern.

    Why Lazy-Load ChromaDB?¶

    Initializing ChromaDB takes ~30 seconds and loads pydantic compatibility patches. Deferring to first use: - Keeps bot startup fast - Only initializes if KB actually used - Still async-safe (concurrent calls wait on first init)

    Why Knowledge Router?¶

    Different query types benefit from different retrieval: - Entity queries (who, where) → graph-first (structured, exact) - Conceptual queries (how, why) → vector-only (semantic, flexible) - Mixed/unknown → hybrid (cover both)

    Router prevents wasting context on irrelevant retrievals.

    Why ChromaDB Over Alternatives?¶

    • Local: No API calls, no rate limits
    • GPU-free: ONNX embeddings work on CPU
    • Persistent: Survives bot restart
    • Query-flexible: Supports various filtering/metadata options

    Glossary¶

    • RAG: Retrieval Augmented Generation — grounding LLM responses in retrieved documents
    • Semantic Search: Finding documents by meaning/relevance, not keyword matching
    • Chunking: Splitting documents into overlapping segments for embedding
    • Entity: A unique concept (person, organization, technology) tracked in the knowledge graph
    • Predicate: A relationship type (works_at, owns, uses) between entities
    • Canonical Form: Normalized version of a name for deduplication
    • TTL: Time-to-Live — how long a cached value remains valid
    • WAL: Write-Ahead Logging — SQLite mode for better concurrent access
