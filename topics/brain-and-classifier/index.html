    FORGE Brain and Classifier Architecture

    10 min read · Last updated April 04, 2026 · 1,918 words

    Summary

    The FORGE brain-and-classifier system routes Discord messages from natural language into executable actions, orchestrating a dual-layer pipeline:

    1. Classifier: models rapidly categorizes messages into ~300 action types with confidence scoring
    2. Brain: models interprets complex requests, loops up to 10 times over tool use, and orchestrates multi-step tasks
    3. Router: Legacy pattern-match fallback (disabled when BRAIN_FIRST=1, the default)

    The architecture prioritizes autonomy: the brain can read Discord channels, post messages, queue tasks for the Copilot agent, and manage its own context state through the brain-state immutable dataclass.

    Classifier System

    Overview

    The classifier operates as a lightweight intent gatekeeper before expensive Claude operations. It answers a single question: "What does the user want to do?" The response is structured JSON with an action field (one of ~300 defined action types) and a confidence score (0.0–1.0).

    Model: models (fastest tier) Speed: ~200ms per classification Cost: ~150 tokens per call Caching: Automatic prompt caching for system messages > 1024 chars

    Action Types

    Actions are defined in core/classifier_prompt.py and cover six categories:

    Category Example Actions Count
    Browser browser_open, browser_screenshot, browser_close, browser_read 6
    System pc_lock, pc_stats, pc_volume, pc_shutdown, pc_restart 8
    Media media_play_pause, media_next, media_mute 5
    Email email_inbox, email_send, email_search, email_mass_delete 12
    Calendar calendar_add, calendar_update, calendar_reschedule 7
    Sourcing prism_score, outreach_draft, candidate_search 15+
    Other Settings, file ops, monitoring, scheduling, recording, etc. ~250

    Classification Process

    When a message arrives:

    1. Message received → Discord fires on_message event in bot.py
    2. Conversational guard → Skip if message is simple yes/no/thanks (no point in classifying)
    3. Pre-classifier intercepts → Hardcoded triggers for frequently-misrouted actions like rc status, nba picks
    4. Classifier call → _classify_action(text) calls models with:
    5. Static system prompt: ~470 action definitions
    6. Dynamic examples: Recent correct classifications from feedback system
    7. Sequence hints: If the user's last action was X, what should they do next?
    8. Context: Last 4 messages in the conversation
    9. Result parsing → JSON extraction from Haiku response
    10. Confidence check → If confidence < 0.5, escalate to brain instead
    11. Dispatch → Route to appropriate cogs or brain

    Confidence System

    The classifier returns a confidence float (0.0–1.0). This value gates:

    • High confidence (>0.8): Execute the action immediately via the cog system
    • Medium confidence (0.5–0.8): Optionally log for retraining
    • Low confidence (<0.5): Escalate to the brain (more expensive, but handles ambiguity)

    Example: "should I email this person?" has low confidence because it's a yes/no question requiring reasoning, not action. The brain takes over.

    Feedback System

    Failed classifications are logged and fed back into future classifications via the feedback system. This creates a self-correcting loop:

    # From core/classifier_feedback.py
dynamic_examples = await classifier_feedback.get_dynamic_examples(limit=10)
# Returns: "User said 'check my email', classifier predicted email_inbox ✓"

    Brain System

    Overview

    The FORGE brain is a persistent Claude Sonnet agent that handles:

    • Complex reasoning and multi-step tasks
    • Tool use loops (up to 10 iterations)
    • Discord channel reading and writing
    • Copilot task queuing
    • Conversation state persistence

    Model: models Max iterations: 10 (tool use loop) Token budget: Configurable per conversation Cost: ~$0.003 per call (vs. ~$0.0001 for classifier)

    Tool Use Loop (10 Iterations)

    The brain's power comes from tool use: Claude can call functions, inspect results, and loop. The ask() method in core/brain.py implements a tool-use loop:

    graph LR
    A[Brain analyzes] --> B{Needs tool?}
    B -->|Yes| C[Call web_search]
    C --> D[Receive results]
    D --> B
    B -->|Yes| E[Call fetch_url]
    E --> F[Receive content]
    F --> B
    B -->|Yes| G[Call forge_write_channel]
    G --> H[Task complete]
    B -->|No| H

    Each iteration:

    1. Claude receives messages + available tools
    2. Claude analyzes and decides: "Do I need a tool?"
    3. If stop_reason == "tool_use", execute all tool calls
    4. Append tool results to messages
    5. Loop back (up to 10 times)
    6. If stop_reason == "end_turn", return text response

    Available Tools

    The brain has access to three tool sets:

    1. FORGE Internal Tools (core/forge_tools.py)

    Always available. Core Discord and task control:

    Tool Purpose
    forge_read_channel Read recent messages from any Discord channel
    forge_write_channel Post a message to Discord
    forge_copilot_task Queue a coding task for the Copilot agent
    forge_edit_message Edit a previously sent message
    forge_delete_message Delete a message
    forge_react Add emoji reaction to a message
    forge_create_thread Create a thread in a channel

    2. Brain Tools (core/brain_tools.py)

    Research and knowledge access:

    Tool Purpose
    web_search Search the web (Serper API)
    fetch_url Extract content from a URL
    search_knowledge Search FORGE's internal ChromaDB knowledge base
    search_candidates Query the candidate evaluation database
    add_candidate Add a new candidate to tracking

    3. MCP Tools (via mcp_manager)

    Optional, dynamically loaded based on available MCP servers. Examples:

    • GitHub API tools
    • Anthropic API tools
    • Email/calendar tools
    • Other integrations

    Brain State (core/brain_state.py)

    Each brain conversation is immutable:

    @dataclass(frozen=True)
class BrainConversation:
    thread_id: str              # Unique ID for this conversation
    user_id: str                # Discord user ID
    channel_id: int             # Discord channel ID
    messages: tuple[dict, ...]  # Full message history
    active_tools: frozenset[str]  # Tools available in this conversation
    cost_accumulated: float     # Total API spend
    facts_injected: tuple[str, ...]  # Additional context injected
    correlation_id: str         # Trace ID for observability
    created_at: str            # ISO 8601 timestamp
    updated_at: str            # Last modification time

    All updates return new frozen instances (immutable pattern):

    conv = conv.with_message({"role": "user", "content": "..."})
conv = conv.with_cost(0.005)  # Add $0.005 to cost

    Persisted to SQLite (data/forge.db, table brain_conversations).

    Token Budget & Caching

    The brain tracks token usage and implements:

    1. Prompt caching: System messages > 1024 chars use Anthropic's ephemeral cache, saving 90% of tokens on repeated calls
    2. Response caching: LRU cache (max 100 responses) for graceful degradation when API fails
    3. Token budgeting: Via modules/token_budget.py, tracks spend per model/source
    4. Circuit breaker: If API fails repeatedly, return cached responses instead of hammering the API

    Classifier Router (Legacy)

    Overview

    When BRAIN_FIRST=0 (disabled by default), FORGE uses the legacy classifier-router pattern defined in core/classifier_router.py.

    Status: Deprecated in favor of brain-first architecture.

    How It Works

    1. Classify via _classify_action() → returns {action, confidence}
    2. Route via _try_action_intent() → dispatch to cog or brain
    3. Execute → cog handles the action synchronously or queues async work

    When It's Still Used

    • Fallback if BRAIN_FIRST=0 in core/config.py
    • Pre-classifier keyword intercepts (hardcoded rules for commonly-misrouted actions)
    • High-confidence classifications route directly to cogs (faster)

    Multi-Action System

    Purpose

    The forge_multi_action action handles compound requests: when the user asks for multiple things at once.

    Example: "Check the talent channel, draft an outreach message, and post it to Slack."

    Flow

    1. User message arrives
    2. Classifier tags it as forge_multi_action
    3. Brain is invoked with Multi-Action Tools
    4. Brain reads channels, composes messages, posts results
    5. All three tasks happen in a single brain call

    Multi-Action Tools

    Subset of FORGE tools, optimized for Discord orchestration:

    • forge_read_channel — read channel history
    • forge_write_channel — post to Discord
    • forge_copilot_task — queue code tasks
    • forge_edit_message — edit sent messages
    • forge_delete_message — clean up
    • forge_react — react with emojis
    • forge_create_thread — start threads

    Bootstrap Sequence

    Overview

    When FORGE starts (bot.py), the core/bootstrap.py initializes all modules in dependency order. Critical failures abort the boot; non-critical failures are logged and skipped.

    Phases

    Phase 1: Core Infrastructure

    graph TD
    A["1. FORGE Monitor<br/>Catches and logs errors"] --> B["2. Process Monitor<br/>Tracks memory, CPU, health"]
    B --> C["3. Database CRITICAL<br/>Create tables, init schema"]
    C --> D{Success?}
    D -->|No| E["Abort"]
    D -->|Yes| F["Continue"]

    Phase 2: JD/Sourcing Filters

    graph TD
    A["4. JD Manager<br/>Load job descriptions"] --> B["5. Sourcing Filters<br/>Set active filters in memory"]

    Phase 3: Classifiers & Models

    graph TD
    A["6. Intent Registry<br/>Load ~300 actions"] --> B["7. Classifier Feedback<br/>Init feedback system"]

    Phase 4: Persistence & State

    graph TD
    A["8. Brain State<br/>Load conversations"] --> B["9. Skill Manager<br/>Load user skills"]

    Phase 5: Skills & Integrations

    graph TD
    A["10. MCP Manager<br/>Connect to MCPs"] --> B["11. Skills Loader<br/>Load from core/skills/"]
    B --> C["12. Cog Loader<br/>Load all cogs"]

    Phase 6: Scheduled Tasks

    graph TD
    A["13. Autonomy Loop<br/>Every 6 hours"] --> B["14. Overnight Engine<br/>8 PM daily"]
    B --> C["15. Sports Pipeline<br/>10 AM daily"]

    Critical vs. Non-Critical

    Critical (fail = abort boot): - Database initialization - Config loading

    Non-critical (fail = log and continue): - Skill loading - Cog loading - MCP connections - Scheduled tasks

    Example: Failed Skill Load

    Module initialize phase... [OK]
Loading skills from core/skills/...
  - skill_search_candidates.py ... [OK]
  - skill_email_draft.py ... [FAIL] (import error)
  - skill_calendar.py ... [OK]
  Database still initialized, other skills work.
  Error logged to system monitor.
  Boot continues.

    Message Flow (on_message)

    Complete Path from Discord to Response

    graph TD
    A[User sends message] --> B[discord.py on_message event]
    B --> C[Parse message]
    C --> D{BRAIN_FIRST=1?}
    D -->|Yes| E[Call brain.ask directly]
    D -->|No| F[Call try_action_intent]
    F --> G[Classify message]
    G --> H[Dispatch to cog or brain]
    E --> I[Brain reads/writes Discord]
    H --> I
    I --> J[Post response to Discord]
    J --> K[Log action to database]

    Brain-First Flow (Default)

    # From bot.py:on_message()
if BRAIN_FIRST:
    response = await brain.ask(
        prompt=message.content,
        system=BRAIN_SYSTEM_PROMPT,
        tools=FORGE_TOOLS + MCP_TOOLS,
        tool_handler=tool_handler,
    )
    await message.reply(response)

    The brain automatically: - Reads/writes Discord - Calls tools as needed - Loops up to 10 times - Returns final response

    Legacy Flow (Disabled by Default)

    # From classifier_router.py:try_action_intent()
intent = await _classify_action(message.content)
# Returns: {"action": "email_send", "confidence": 0.87}

if intent["confidence"] > 0.8:
    # Fast-path to cog
    cog = dispatch_to_cog(intent["action"])
    await cog.execute(message, intent)
else:
    # Escalate to brain
    await brain.ask(message.content, ...)
