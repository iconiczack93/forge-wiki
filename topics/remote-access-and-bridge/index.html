    Remote Access & Bridge

    4 min read · Last updated April 04, 2026 · 745 words

    FORGE's mobile access and remote Claude Code integration.

    Remote Control: Mobile Session Access

    Spawns claude remote-control CLI for secure remote access from phone/tablet.

    What It Does

    • Runs Claude Code as a plain (non-TUI) process
    • Prints session URL to stdout
    • Stays alive until killed
    • Generates handoff file on stop with git context

    Starting Remote Control

    "start rc"  →  Launches Claude Code remote-control session
             →  Returns session URL for mobile/tablet access

"night mode"  →  Alias for RC (Zack uses at night)

"rc status"  →  Shows current RC session URL, uptime

"stop rc"  →  Kills RC, generates handoff MD with git state

    Environment Handling

    Strips all CLAUDE_*, ANTHROPIC_*, and CLAUDECODE vars before launch. This forces Claude Code to use Max subscription OAuth (not API billing).

    Critical vars cleared: - CLAUDE_CODE_OAUTH_TOKEN (overrides stored credentials) - CLAUDE_CODE_ENTRYPOINT (breaks RC detection)

    Handoff Generation

    On RC stop, writes to .claude/handoffs/: 

    rc_handoff_{timestamp}.md
  ├── git status
  ├── git diff (all changes)
  ├── modified files list
  ├── session metadata (start time, duration)
  └── next-session context

    Next local Claude Code session reads this handoff for full context.

    Configuration

    Setting Value Purpose
    CLAUDE_CMD C:\nvm4w\nodejs\claude.cmd Claude Code CLI path
    HANDOFF_DIR .claude/handoffs/ Handoff file location
    _RC_PROCESS subprocess.Popen Managed process handle

    Mobile Bridge: Claude Code on Your Phone

    Self-hosted web bridge (Starlette) spawns Claude Code CLI in --print stream-json mode, streams responses to mobile chat UI over WebSocket.

    Architecture

    Phone (WebSocket client)
    ↓
Bridge (Starlette server, port 8080)
    ↓
Claude Code CLI (--print stream-json)
    ↓
max_token_limit response stream

    Usage

    Start from Discord: 

    "start bridge"  →  Launches on port 8080
                →  Returns: http://your-ip:8080/session-token

"bridge status"  →  Current URL, active sessions

"stop bridge"  →  Shuts down cleanly

    Standalone (no Discord): 

    python modules/bridge.py --port 8080

    Features

    • Session Manager: tracks concurrent Claude Code sessions (max 10 default)
    • WebSocket streaming: real-time message exchange
    • Auth token: random token per session (URL includes token)
    • Idle cleanup: removes sessions after inactivity
    • Preview text: shows first 100 chars of each session in UI

    Mobile UI

    Lightweight HTML5 dashboard: - List active sessions - Create new session - Real-time chat with streaming responses - Message history per session

    Configuration

    PORT = 8080
MAX_SESSIONS = 10
SESSION_IDLE_TTL = 3600  # 1 hour inactivity

_server_process = None  # managed subprocess
_server_url = None      # broadcast to Discord
_auth_token = None      # random per session

    Data Classes

    SessionInfo: 

    internal_id: str       # UUID (first 8 chars)
claude_session_id: Optional[str]
created_at: float
last_active: float
preview: str           # message preview

    RC Status & Health

    Monitor RC/Bridge uptime and activity from Discord.

    Command Info
    rc status RC session URL, uptime, process health
    bridge status Bridge URL, active sessions, port
    night mode Start RC for mobile access
    mobile claude Alias for bridge

    Security & Auth

    • RC: Uses Max OAuth (no API key exposure)
    • Bridge: Random token per session, WebSocket over HTTPS (if deployed)
    • Handoff: Sensitive git state only written to local .claude/handoffs/

    Best Practices

    1. Never share RC/Bridge URLs publicly
    2. RC token is embedded in handoff file—keep .claude/handoffs/ private
    3. Bridge auth token regenerates per session
    4. Kill RC when done (handoff context is one-way, not continuous)

    Key Files

    File Purpose
    modules/remote_control.py RC launcher, process manager, handoff generator
    modules/bridge.py Bridge server, WebSocket handler, session manager
    modules/bridge_ui.html Mobile chat UI (embedded in bridge)
    cogs/remote_control_cog.py Discord command handlers (start, stop, status)
    cogs/bridge_cog.py Discord bridge handlers
    .claude/handoffs/ Session context files (git state, changes)

    Example Workflow

    Mobile coding session at night:

    Zack (on phone): "start bridge"
FORGE: "Bridge running on http://192.168.1.X:8080/session-token"
       → Zack opens URL on phone → Claude Code chat UI loads
       → "add dark mode toggle to finance dashboard"
       → Claude Code (via bridge) reads code, makes changes, streams response
       → "done, pushed to main"

Next morning (local machine):
FORGE reads .claude/handoffs/bridge_*.md
→ Full context: what was changed, git state, session duration
→ Local Claude Code picks up exactly where mobile left off
