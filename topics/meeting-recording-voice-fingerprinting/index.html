    Meeting Recording and Voice Fingerprinting System

    8 min read · Last updated April 04, 2026 · 1,686 words

    Summary

    FORGE's meeting recording system captures audio (system output and microphone) from Windows WASAPI, transcribes conversations with AssemblyAI's speaker diarization, and automatically maps speakers (Speaker A/B/C) to real names using voice embeddings and Claude-powered contextual analysis. The system runs as a decoupled standalone process that survives FORGE crashes, learns speaker identities over time via voice fingerprinting, and produces structured meeting analysis with action items grouped by owner.

    Architecture Overview

    The meeting recording system consists of six major components working together:

    graph TD
    A["FORGE Bot<br/>Routes /record-start<br/>Manages post-pipeline<br/>Stores context"] --> B["Control File IPC<br/>recorder_control.json"]
    B --> C["forge_recorder.py<br/>Standalone Process<br/>Ring buffer 300s<br/>WASAPI + mic capture<br/>Survives crashes"]
    C --> D["Audio Files<br/>system.wav<br/>mic.wav"]
    D --> E["Post-Recording Pipeline"]
    E --> F["1. Merge + compress"]
    F --> G["2. AssemblyAI transcribe"]
    G --> H["3. Extract embeddings<br/>Resemblyzer"]
    H --> I["4. Speaker resolution<br/>Claude"]
    I --> J["5. Analyze + action items"]
    J --> K["6. Store KB + Discord"]

    Control Flow: Inter-Process Communication

    1. User command: /record-start "Meeting Title" in Discord
    2. Bot writes control file: data/recorder_control.json with {"action": "start", "meeting": "Meeting Title"}
    3. Standalone recorder polls the control file, sees start command
    4. Recorder launches capture threads, starts ring buffer
    5. Recorder writes lockfile: data/recordings/.active_recording.json with active PID
    6. User command: /record-stop
    7. Bot writes stop command to control file
    8. Recorder stops capture, writes result to data/recorder_result.json
    9. Bot reads result, launches transcription pipeline

    This IPC design means the recorder can survive FORGE crashing—even if the bot dies, the recorder keeps capturing until /record-stop is called or 1 hour elapses (hard limit).

    Voice Fingerprinting

    How It Works

    Voice fingerprinting uses Resemblyzer, a neural speaker embedding model that converts audio segments into 256-dimensional vectors representing unique speaker characteristics.

    Process: 1. During transcription, AssemblyAI segments audio by speaker (Speaker A, Speaker B, etc.) 2. For each segment, extract a 3-5 second audio clip 3. Resample to 16kHz (Resemblyzer's expected sample rate) 4. Pass through VoiceEncoder model → 256-dim embedding vector 5. Compare new embedding against stored speaker profiles using cosine similarity 6. If similarity > 0.75 (threshold), match to known speaker; otherwise flag as new speaker

    Threshold and Tuning

    • Similarity threshold: 0.75 (cosine distance)
    • Sample rate: 16kHz (Resemblyzer requirement)
    • Embedding dimension: 256
    • Model: Pre-trained speaker recognition encoder from SpeechBrain

    The threshold was chosen to balance false positives (matching wrong speakers) against false negatives (missing known speakers). High-confidence matches (>0.85) are auto-accepted; borderline matches (0.75-0.85) are flagged for human review.

    Speaker Profile Database

    Profiles are stored in forge.db in three related tables:

    speaker_fingerprints (per-utterance): - speaker_name — name of known speaker - embedding — 256-dim vector (JSON) - utterance — text of confirmed utterance - confidence — human confidence (0.0-1.0)

    speaker_behavioral_stats (aggregated): - speaker_name - filler_rate — fillers per 100 words (um, uh, like, etc.) - question_rate — % of utterances that are questions - avg_utterance_length — mean words per utterance - common_phrases — distinctive 2-3 word phrases - sample_utterances — 3 most recent confirmed utterances - sample_size — total confirmed utterances

    calibration_sessions (per-meeting accuracy): - meeting_date - total_speakers — how many unique speakers detected - matched_speakers — how many had names resolved - accuracy_score — (matched / total)

    Active Learning

    The fingerprint database improves over time:

    1. Calibration: After initial speaker resolution, prompt user to verify speakers in Discord
    2. Correction: User runs /fix-speaker "Speaker A" "John" → stores utterances as confirmed
    3. Behavioral profiling: System analyzes confirmed utterances to extract filler rate, vocabulary patterns
    4. Next meeting: New speakers are compared against behavioral profiles and embeddings together

    This two-stage approach catches speakers even if their voice changes (cold, stress, etc.) by also matching speaking patterns.

    Auto-Recording

    calendar_auto_record.py provides calendar-based meeting detection:

    Config: Stored in data/meeting_auto_record.json: 

    {
  "enabled": true,
  "meetings": [
    {"pattern": "Eng Sync", "expected_speakers": ["Zack", "Arda", "Hank"]},
    {"pattern": "all-hands", "expected_speakers": ["Zack", "Arda", "Linda", "Hank"]}
  ]
}

    How it works: 1. Periodic job (runs every 5 min) checks Google Calendar for next 30 minutes of events 2. For each event, check if event title contains any configured pattern (case-insensitive substring match) 3. If matched and meeting hasn't already been recorded today, auto-start recorder 4. If expected_speakers is provided, pass count to AssemblyAI as hint for diarization

    Opt-out: Recurring meetings are stored in data/meeting_recorded_events.json to avoid re-recording same meeting twice in one day.

    Speaker Resolution

    speaker_resolution.py maps Speaker A/B/C to real names using a three-stage process:

    Stage 1: Context Building

    Gather hints from: - Me: Current user (Zack), role, notes - Contacts: Known colleagues (Arda, Linda, Hank, etc.), roles, notes - Recurring meetings: Day-of-week patterns ("Eng Sync is always Monday with Zack, Arda, Hank") - Ambiguity hints: Common confusion pairs ("If there are only 2 speakers, Speaker B is likely Hank, not Arda")

    Stage 2: Behavioral Comparison

    For each detected speaker: 1. Extract voice embedding from sample utterances 2. Compare against stored speaker fingerprints using cosine similarity 3. If high confidence match (>0.85), propose name 4. Combine with behavioral stats (filler rate, question patterns)

    Stage 3: Claude Arbitration

    Send to Claude: - Context hints (who's likely in this meeting) - Transcription samples (first 2-3 utterances per speaker) - Embedding distances (if fingerprints exist: "Speaker A is 0.88 similar to Hank, 0.71 to Arda") - Behavioral signals (Speaker B has 12% filler rate → matches Arda profile)

    Claude synthesizes all signals and returns structured mapping: 

    {
  "Speaker A": "Zack",
  "Speaker B": "Arda",
  "Speaker C": "?Unknown",
  "confidence": {"Speaker A": 0.95, "Speaker B": 0.82, "Speaker C": 0.0}
}

    Unknown speakers are flagged for manual calibration.

    Post-Processing Pipeline

    After recording stops, the system runs a sequential pipeline:

    1. Audio Merge & Compression

    • Read system.wav (system output) and mic.wav (microphone)
    • Mix both into a single stereo WAV (left=system, right=mic)
    • Compress to MP3 (128kbps) to save space
    • Store as recordings/{date}_{meeting_title}.mp3

    2. Transcription (AssemblyAI)

    • Upload MP3 to AssemblyAI API
    • Request speaker diarization with speaker_labels=True
    • Optionally pass speaker count hint from calendar if known
    • Poll for completion (typically 1-3 minutes for 1-hour meeting)
    • Result includes speaker labels (Speaker A, Speaker B, etc.) and timestamps

    3. Voice Fingerprinting

    • For each unique speaker label, extract 5-second audio clip at midpoint of their longest utterance
    • Resample to 16kHz
    • Compute 256-dim embedding vector
    • Compare against stored profiles in speaker_fingerprints table
    • Output similarity scores for top 3 matches per speaker

    4. Speaker Name Resolution

    • Use speaker_resolution.py to map labels to names
    • Store mapping in metadata file: recordings/{meeting_id}_speakers.json
    • Flag any unknown speakers for human verification

    5. Analysis & Action Items

    • Replace generic Speaker A/B/C with resolved names in transcript
    • Send full transcript to Claude
    • Extract: TL;DR, action items (grouped by owner), decisions made
    • Store analysis in recordings/{meeting_id}_analysis.json

    6. Knowledge Base Ingestion

    • Add transcript + analysis to FORGE's vector KB (ChromaDB)
    • Chunk by speaker turns for precise retrieval
    • Index by date, attendees, keywords

    7. Discord Output

    • Post summary to #meeting-recordings (if configured)
    • Include: title, attendees, TL;DR, action items, link to full transcript

    Known Issues and Limitations

    "Stuck Typing" Post-Pipeline Hang

    Symptom: After transcription completes, speaker resolution appears to hang for 2-5 minutes. No errors logged. Claude calls seem blocked.

    Root cause: Resemblyzer's VoiceEncoder model loads lazily (first time used). Loading the ~100MB model from disk on first embedding extraction blocks the event loop.

    Workaround: Pre-load encoder in bootstrap.py by calling _get_encoder() once on bot startup. Or move embedding extraction to background thread pool.

    Status: Identified, not yet fixed. Consider batching embedding computation or async subprocess call.

    Recorder Crash-Loop (Historical)

    Previous issue (now fixed): Recorder thread crashed on WASAPI initialization → lockfile not cleared → next start attempt read stale PID → think recording already active.

    Fix applied: - Moved audio capture to standalone forge_recorder.py with independent lifecycle - Ring buffer survives FORGE crashes - Supervisor process auto-restarts recorder if it dies - Lockfile includes timestamp + PID validation

    AssemblyAI Speaker Count Hints

    Limitation: Passing expected_speakers count to AssemblyAI improves diarization accuracy only if count is exact. If you hint 3 speakers but there are actually 2, AssemblyAI may hallucinate a third speaker with ~30% silence.

    Mitigation: Use expected_speakers only when you're confident (calendar shows exact attendee count, or recurring meeting with fixed attendees).

    Voice Similarity on Noisy Calls

    Known issue: On calls with high background noise or poor connection, Resemblyzer embeddings can drift, causing similarity scores to drop below 0.75 even for known speakers.

    Mitigation: - Fallback to behavioral profile matching (filler rate, question rate) - Manually verify speakers in Discord after each meeting (/fix-speaker "Speaker A" "John") - Calibration feedback improves fingerprints for future meetings

    Large Meeting Performance

    Limitation: For meetings with 5+ speakers, speaker resolution becomes ambiguous. Claude's contextual hints help, but absent explicit "who's who" cues, accuracy degrades.

    Recommendation: For large meetings, pre-configure speakers in meeting_speakers.json contacts list with detailed notes.
