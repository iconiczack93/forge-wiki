    Discord Bot Infrastructure

    13 min read · Last updated April 04, 2026 · 2,599 words

    FORGE is a sophisticated Discord bot built on discord.py that orchestrates AI-powered workflows, sourcing pipelines, and real-time progress tracking. This article covers the core architectural components that make FORGE function across 6,698 lines of bot logic, 34 cogs, and 38+ modules.

    Overview

    FORGE operates on a dispatcher + event-driven architecture:

    • Classifier: Routes natural language → ~300 action types
    • Cog System: 34 specialized cogs handle domain-specific logic (sourcing, lab, admin, views, etc.)
    • Event Bus: Lightweight pub/sub for cross-module causation tracking
    • Middleware Chain: Pre/post-processing around Claude API calls
    • Progress Tracker: Real-time Discord message updates for long-running tasks
    • Channel Manager: Unified interface for reading/writing Discord messages

    This design separates concerns while maintaining a single message handler in bot.py that stays clean and testable.

    Bot Initialization & Message Routing

    File: [[bot.py]]

    Duplicate Instance Guard

    FORGE prevents two bot.py processes from running simultaneously using PID tracking:

    bot_pid.json → stores PID + startup timestamp
_check_bot_duplicate() → verifies process is still alive before allowing startup

    The guard uses psutil to validate: 1. Process exists and is Python 2. Process is running bot.py (not a recycled PID)

    Core Message Handler

    The main on_message event processes every Discord message through:

    1. Ignore checks: Skip bot messages, private DMs, empty content
    2. Relay check: If Claude-app relay is active, intercept and forward
    3. Classifier: Natural language → action type
    4. Dispatcher: Action type → cog handler via HANDLED_ACTIONS
    5. Brain call (optional): Invoke Claude for multi-action workflows
    6. Cog execution: Async task execution with error handling

    Configuration

    File: core/config.py

    Environment-based settings include:

    DISCORD_BOT_TOKEN           # Bot authentication
DISCORD_GUILD_ID            # Primary guild
DISCORD_CHANNEL_ID          # Default channel (⚡-forge)
MODELS                      # Haiku, Sonnet, Opus endpoints
API_KEYS                    # Anthropic, Tavily, Exa, Notte, etc.
LAB_CHANNEL_ID              # 💡-lab (autonomous brainstorming)
SCOUT_CHANNEL_ID            # Sourcing scout operations
MONITOR_CHANNEL_ID          # 📡-monitor (error detection)
COPILOT_CHANNEL_ID          # 🎮-copilot (Claude↔Discord bridge)

    All config is lazy-loaded to support hot-reload of values.

    Cog System (HANDLED_ACTIONS)

    File: cogs/init.py (typically minimal)

    Each cog is a commands.Cog subclass that:

    1. Defines one or more actions via HANDLED_ACTIONS dict
    2. Implements handle_action(action, **kwargs) async method
    3. Lives in cogs/ directory and is auto-loaded at startup

    HANDLED_ACTIONS Pattern

    # In a cog:
HANDLED_ACTIONS = {
    "retrieve_jd": {
        "description": "Fetch job description from Ashby",
        "params": ["job_id"],
    },
    "evaluate_candidate": {
        "description": "Run PRISM evaluation",
        "params": ["profile_text", "score_threshold"],
    },
}

async def handle_action(self, action: str, **kwargs):
    if action == "retrieve_jd":
        return await self._fetch_jd(kwargs["job_id"])
    elif action == "evaluate_candidate":
        return await self._prism_evaluate(kwargs["profile_text"], kwargs["score_threshold"])

    Built-in Cogs

    Cog Purpose Key Actions
    admin.py Owner-only commands reload, reload_all, sync_commands, cogs
    views.py Persistent UI panels ConfirmView, DestructiveConfirmView, ScoutControlView
    sourcing_cmds.py Candidate sourcing search_linkedin, enrich_profile, score_candidate
    lab.py Autonomous brainstorming agent_run, agent_status, agent_output
    tts_actions.py Text-to-speech pipeline speak, stream_audio
    ~30 more Specialized domains Valorant, PrizePicks, enrichment, monitoring, etc.

    Hot-Reload

    Admin commands allow live cog reloading without restarting the bot:

    /reload sourcing_cmds  # Reloads just sourcing_cmds cog
/reload_all            # Reloads all cogs
/sync_commands         # Re-syncs slash commands to Discord

    Event Bus

    File: core/event_bus.py — Holy Grail #9

    A lightweight async pub/sub system for cross-module causation tracking.

    Key Features

    • Typed Events: Dataclass-based events with correlation IDs
    • Causation Chains: Parent event IDs track call hierarchy
    • SQLite Persistence: Events survive crashes for analytics
    • Bounded Queue: Prevents backlog (MAX_QUEUE = 500)
    • Subscriber Isolation: One bad handler doesn't crash others

    Event Dataclass

    @dataclass
class Event:
    event_type: str              # "candidate_found", "brain_call_completed", etc.
    data: dict                   # Payload
    source_module: str           # "pipeline_scout", "middleware.cost_tracking"
    event_id: str                # Auto-generated UUID
    parent_event_id: Optional[str]   # Links to parent (causation)
    correlation_id: str          # Groups related events
    timestamp: str               # ISO 8601

    Usage

    from core.event_bus import event_bus

# Emit an event
event_id = await event_bus.emit(
    "candidate_found",
    {"name": "Jane", "source": "github", "score": 4.2},
    source_module="pipeline_scout",
    parent_event_id=strategy_event_id,
)

# Subscribe to events
@event_bus.on("candidate_found")
async def handle_candidate(event):
    print(f"New candidate: {event.data['name']}")

# Query causation chain (root → leaf)
chain = await event_bus.get_chain(event_id)

# Get all events in a correlation group
group = await event_bus.get_correlation_group(correlation_id)

# Get descendants (recursive)
descendants = await event_bus.get_descendants(event_id)

    Middleware Integration

    The event bus is automatically wired into #Middleware Chain:

    • EventBusMiddleware: Emits brain_call_started / brain_call_completed events
    • CostTrackingMiddleware: Emits brain_cost_tracked with token counts and pricing

    Middleware Chain

    File: core/middleware_chain.py

    Inspired by Deer-Flow, adapted for FORGE's immutable-first design.

    Architecture

    Middlewares are applied in onion model:

    graph LR
    A["Input"] --> B["mw[0]"]
    B --> C["mw[1]"]
    C --> D["mw[2]"]
    D --> E["Claude API"]
    E --> F["mw[2]"]
    F --> G["mw[1]"]
    G --> H["mw[0]"]
    H --> I["Output"]

    Core Data Types

    @dataclass(frozen=True)
class BrainContext:
    """Immutable snapshot before Claude API call."""
    messages: tuple[dict, ...]
    system_prompt: str
    user_id: str
    channel_id: int
    model: str
    tools: tuple[dict, ...]
    correlation_id: str
    extra: dict

@dataclass(frozen=True)
class BrainResponse:
    """Immutable snapshot after Claude API call."""
    content: str
    tool_calls: tuple[dict, ...]
    usage: dict
    cost: float
    stop_reason: str

    Built-in Middlewares

    EventBusMiddleware

    Emits lifecycle events around Claude calls:

    • brain_call_started: Before API call (includes message count, tool count)
    • brain_call_completed: After API call (includes duration, stop_reason, tool_call_count)

    Tracks start times per correlation_id to calculate duration.

    CostTrackingMiddleware

    Logs cost and usage tokens:

    Brain call cost: $0.002345 (in=512, out=128, model=claude-sonnet-4-5)

    Emits brain_cost_tracked event for centralized cost aggregation.

    SubagentLimitMiddleware

    Prevents runaway forge_copilot_task tool spawning:

    • Default: max 3 copilot calls per brain response
    • Excess calls are truncated with a warning
    • Helps prevent token explosion from nested subagent spawning

    Usage in FORGE

    from core.middleware_chain import default_chain, BrainContext, BrainResponse

chain = default_chain()

# Before call
ctx = BrainContext(
    messages=tuple(messages),
    system_prompt=system,
    user_id=str(author.id),
    channel_id=channel.id,
    model=model_id,
    tools=tuple(tools),
    correlation_id=correlation_id,
)
ctx = await chain.run_before(ctx)

# Make Claude API call
response = await client.messages.create(...)

# After call
resp = BrainResponse(
    content=response.content[0].text,
    tool_calls=tuple(tool_dicts),
    usage={"input_tokens": response.usage.input_tokens, ...},
    cost=calculated_cost,
    stop_reason=response.stop_reason,
)
resp = await chain.run_after(ctx, resp)

# Wrap tool execution
result = await chain.run_tool("forge_read_channel", args, base_handler)

    Progress Tracking

    File: core/progress.py

    Universal progress display for long-running operations. Shows real-time status in Discord via throttled message edits.

    ProgressTracker

    tracker = ProgressTracker(
    channel=channel,
    title="PRISM Evaluation",
    edit_interval=1.5,  # Throttle edits to avoid rate limits
    show_elapsed=True,
    show_spinner=True,
    max_display_steps=8,
)

async with tracker:
    await tracker.step("Loading JD...")
    # ... do work ...

    await tracker.step("Evaluating candidate...")
    await tracker.substep("Checking tech stack")
    # ... do work ...

    await tracker.complete("Score: 4.2/5")

    🔄 **PRISM Evaluation**
✅ Loading JD...
✅ Fetching GitHub history...
⠋ Evaluating candidate...
   └ Checking tech stack
⏱️ 3.2s

    When finished:

    ✅ **PRISM Evaluation**
✅ Loading JD...
✅ Evaluating candidate...

Score: 4.2/5
⏱️ 12.4s

    Streaming Support

    For Claude responses, use on_token callback:

    tracker = ProgressTracker(channel, title="Thinking")
async with tracker:
    response = await brain.ask_stream(
        messages=messages,
        on_token=tracker.stream_text,
    )

    Shows last ~200 chars of streaming text as preview.

    Progress Bar

    await tracker.progress_bar(current=40, total=50, label="Enriching...")
# [████████░░] 80% (40/50) Enriching...

    NoOpTracker

    For optional progress tracking, use NoOpTracker:

    from core.progress import get_tracker

async def long_operation(tracker=None):
    tracker = get_tracker(tracker)
    await tracker.step("Starting...")
    # ... now safe to call tracker even if None

    Claude↔Discord Relay

    File: modules/relay.py

    Enables Claude (from the Claude desktop app) to post to #copilot and receive Zack's replies via Discord MCP without spawning the copilot CLI.

    How It Works

    1. Claude posts to #copilot via Discord MCP
    2. FORGE detects Claude² bot message
    3. If message has attention prefix, FORGE DMs Zack
    4. Zack replies in #copilot (from phone, etc.)
    5. Claude reads reply via Discord MCP (no CLI spawn)
    6. Session auto-times out after 15m idle

    Attention Prefixes (trigger DM)

    🔔 Need input: [question]
❓ Should we do X?
⚠️ Found a problem: [details]
❌ Build failed
✅ Task complete
🏁 Session done

    Status Prefixes (no DM)

    📋 Working on X...
🔧 Editing file Y
📝 Found 3 issues
⏳ In progress

    Usage

    from modules.relay import handle_relay_message, get_relay_status

# In on_message handler
if await handle_relay_message(message):
    return  # Message handled, skip copilot CLI

# Get relay status
status = get_relay_status()
# "🟢 **Relay active** (5m) — 12 messages"
# "⚫ **Relay idle** — post to #copilot to start"

    Session Control

    In #copilot, Zack can:

    end relay        # End relay, restore copilot CLI
stop relay       # (alias)
copilot mode     # (alias)

    Implementation

    @app_commands.command(name="reload")
async def cmd_reload(self, interaction: discord.Interaction, cog: str):
    if not self._is_owner(interaction):
        await interaction.response.send_message("Owner only.", ephemeral=True)
        return

    ext = f"cogs.{cog}"
    try:
        await self.bot.reload_extension(ext)
        await interaction.response.send_message(f"Reloaded `{ext}`.")
    except Exception as e:
        await interaction.response.send_message(f"Failed: {e}")

    All admin commands are owner-guarded via _is_owner() check.

    Interactive Views

    File: cogs/views.py

    Persistent button-based UI panels using discord.py views.

    ConfirmView

    Simple yes/no confirmation:

    view = ConfirmView(author_id=ZACK_ID, timeout=60)
msg = await channel.send("Proceed with enrichment?", view=view)

if await view.wait():  # Wait for interaction
    if view.value:
        await channel.send("✅ Confirmed")
    else:
        await channel.send("❌ Cancelled")

    DestructiveConfirmView

    Confirmation with countdown timer for destructive actions:

    view = DestructiveConfirmView("Restart FORGE", timeout=12)
msg = await channel.send("⚠️ **Restart FORGE?** This will interrupt tasks.", view=view)
await view.start_countdown(msg)

if await view.wait():
    if view.value:
        # User confirmed after countdown
    else:
        # Timed out or cancelled

    Countdown runs 10 → 8 → 6 → 4 → 2 seconds on the button label.

    ScoutControlView

    Sourcing scout operations (run, stats, autopilot):

    • Button-based control for pipeline operations
    • Persistent across restarts (custom_id + timeout=None)
    • Integrates with #Progress Tracking for status updates

    Implementation Pattern

    class MyView(discord.ui.View):
    def __init__(self):
        super().__init__(timeout=None)  # Persistent
        self.value = None

    @discord.ui.button(label="Confirm", style=discord.ButtonStyle.green)
    async def confirm_btn(self, interaction: discord.Interaction, button: discord.ui.Button):
        if interaction.user.id != ZACK_ID:
            await interaction.response.send_message("Not authorized", ephemeral=True)
            return

        self.value = True
        self.stop()
        await interaction.response.edit_message(view=None)

    All views use custom_id for persistent message handling across bot restarts.

    Architecture Diagram

    graph TD
    A["Discord Message"] --> B["bot.py: on_message"]
    B --> C["Ignore checks"]
    C --> D["Relay check"]
    D --> E["Classifier<br/>NLP to action"]
    E --> F["HANDLED_ACTIONS<br/>dispatcher"]
    F --> G["Cog handler"]
    G --> H["Progress tracking<br/>ProgressTracker"]
    G --> I["Claude API calls<br/>Middleware chain"]
    I --> I1["EventBusMiddleware"]
    I --> I2["CostTrackingMiddleware"]
    I --> I3["SubagentLimitMiddleware"]
    G --> J["Channel I/O<br/>discord_channels"]
    K["Event Bus<br/>cross-module pub/sub"] -.->|brain_call_started| L["100+ event types"]
    K -.->|brain_call_completed| L
    K -.->|brain_cost_tracked| L
    K -.->|candidate_found| L
    M["Admin Commands<br/>/reload /sync_commands"] --> N["Hot-reload cogs<br/>no restart"]

    Design Principles

    1. Immutability: All context/response objects are frozen dataclasses
    2. Causation tracking: Event bus links parent→child relationships
    3. Isolation: Middleware failures don't crash the chain
    4. Throttling: Progress tracker batches Discord edits (default 1.5s)
    5. Lazy loading: Config values fetched on-demand for hot-reload support
    6. Observer pattern: Cogs register actions at startup, dispatcher invokes
    7. Error recovery: Graceful degradation for Discord API failures
