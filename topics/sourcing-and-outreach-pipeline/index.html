    FORGE Sourcing and Outreach Pipeline¶

    Summary¶

    The FORGE sourcing and outreach pipeline is a multi-stage, AI-driven system that discovers engineering candidates from external sources (GitHub, Slack communities, company raids), enriches their LinkedIn profiles, scores them using PRISM evaluation, and generates personalized outreach messages. The pipeline is orchestrated via [[Sourcing Pipeline]] (4-agent orchestrator) and can run on-demand or automatically overnight via [[Overnight Engine]].

    Core flow: 

    [Strategist] → [Scout] → [Enricher] → [Evaluator] → [Outreach] → [Ashby/Tracking]
   (Plan)     (Discover)  (Enrich)    (Score 1-5)  (Personalize) (Pipeline)

    Each stage is independently parallelizable, allowing the pipeline to scale across multiple concurrent discovery streams (GitHub, Slack communities, company raids) while maintaining cost control and quality gates.

    Company Raids¶

    [[Company Raider]] executes a two-phase discovery targeting engineers at specific companies:

    Stage 1: Apify Batch Search - Query LinkedIn by company name and experience bracket (typically 2-7yr and 6-10yr) - CRITICAL: Apify calls are always batched — never one profile at a time (prevents $200+ waste on setup fees) - Batch size: 200 profiles per flush - Cost: ~$0.003/profile via Apify batch API

    Stage 2: Post-Filter & Dedup - Filter by job title (ENGINEER_KEYWORDS: "engineer", "ml", "ai", "backend", etc.) - Remove executive roles (VP, director, founder) - Geographic filter: Bay Area focus (SF, Palo Alto, Mountain View, etc.) - Experience validation: 2-7yr junior roles flagged as "senior_backlog" if 8+yr - Dedup against: - Ashby pipeline (candidates already in ATS) - candidate_evals table (already scored) - profile_scrape_cache (already scraped today)

    Output: - Qualified candidates forwarded to PRISM scorer - Senior overqualified candidates tagged for future hiring rounds

    Data files: - Target companies: data/ai_companies_master.csv - Bay Area city list: hardcoded in raider - Blocklist: COMPANY_BLOCKLIST (Google, Meta, Amazon, etc.)

    LinkedIn Enrichment¶

    LinkedIn profile enrichment happens in two flavors:

    RapidAPI (Primary, Lower Cost)¶

    [[linkedin_ops.py|linkedin_ops.py]] integrates with fresh-linkedin-profile-data.p.rapidapi.com: - Pulls public profile data (education, experience, skills) in JSON - Cost: ~$0.006/profile (lower than Apify, but less detailed) - Used for quick screening when full Apify scrape not justified - Batch requests allowed — parallelize up to 10 concurrent

    Never use Apify for LinkedIn. ([[Memory#Rules|see MEMORY rules]]) — prior usage wasted $200+ on per-call overhead.

    Apify Batch Scraping (Fallback)¶

    [[pipeline_enricher.py|pipeline_enricher.py]] uses supreme_coder~linkedin-profile-scraper actor: - Full LinkedIn profile extraction: headline, experience, education, skills, endorsements, recommendations - 200 profiles per batch to amortize setup cost (~$5 overhead per run) - Timeout: 5 minutes before batch flushes - Cost: ~$0.003/profile (cheaper per-unit than RapidAPI, but requires batching)

    Pre-filtering via Haiku: - Before Apify batch, all URLs get Haiku pre-filter (JD-relevance check) - Helps drop obvious non-matches before expensive scraping - Improvement: 30-40% of discovered URLs filtered before Apify

    Post-filtering: - Full profiles checked against sourcing_filters (years exp, location, company) - Failing profiles dropped before scoring

    Storage: - Raw profiles stored in pipeline_profiles table - Cached indefinitely; dedup prevents re-scraping same URL within cooldown window

    PRISM Scoring¶

    [[pipeline_evaluator.py|pipeline_evaluator.py]] runs PRISM scoring on enriched profiles using Claude Sonnet.

    Score Scale (1-5): - 5: Strong match — multiple must-haves met, relevant tech stack, seniority aligned, geographic fit - 4: Good match — most must-haves, solid background, some gaps okay (nice-to-haves) - 3: Possible match — core requirements met, but questions around depth/breadth - 2: Weak match — missing 2-3 key requirements, requires significant backfill - 1: Not a fit — fundamentally misaligned (e.g., frontend-only for backend role)

    Input: Enriched LinkedIn profile + JD content (title, requirements, seniority, tech stack)

    Output: Scored record with: - Score (1-5) - Reasoning (why this score) - Timing signal (hire urgently vs. backlog vs. not now) - Outreach context (what to emphasize)

    Batch processing: 10 profiles at a time to maximize Sonnet throughput, minimize latency overhead

    Recent layoff flagging: Candidates from Meta, Amazon, Google, Microsoft, Salesforce, etc. flagged as potentially motivated (recent layoff → active job search)

    Database: Stored in candidate_evals table with full PRISM report

    Outreach Generation¶

    [[linkedin_ops.py|linkedin_ops.py]] and [[Ashby Integration|Ashby ATS]] feed outreach generation.

    Quality Gate: 25-Rule Evaluator All outreach runs through rules check before showing to Zack:

    Hard Bans (Fix before posting): - NO em dashes (—) anywhere. Use commas, periods, parentheses instead. - NO "Worth a chat/conversation/call/anything" — banned phrase - NO "on your radar" — overused - NO "I came across your profile" or "I was impressed by" - NO filler ("That's a unique combo", "That's exactly the kind...") - NO restating candidate's resume back to them - NO time commitments ("15 min", "quick call") - NO claims about HockeyStack unless verified by Arda - NO claim of contractor status without verification

    Format Rules: - 1-2 sentences MAX - :) only closes observation sentence, never mid-sentence - Blurb only — no "Hi [Name]" or "Arda" wrapper (Ashby template handles that) - NO CTA as final line — personalization is the ask. Observation closes it. - Sample CTAs: "Happy to share more if you're curious." / "Open to chatting if you are." / Ask a genuine question instead.

    Style (Arda's Voice): - Casual, direct, no corporate fluff - Show you read their profile — reference specific project or skill - Personal observation at end: "Your [specific thing] stands out for [why it matters]" - Never compare to other candidates - One at a time — never bulk templates

    Example (good):

    Your work on distributed systems at Stripe + move to Cursor seems like you're betting big on AI-first tooling. That's exactly the vibe we're building at HockeyStack :)

    Example (bad):

    Hey John, I came across your profile and was impressed by your background. I think we have an interesting opportunity that might be worth a conversation. Let me know if you're open to a quick 15-min call!

    Sourcing Pipeline Orchestration¶

    [[sourcing_pipeline.py|sourcing_pipeline.py]] coordinates the 4-agent pipeline:

    Agent 1: Strategist¶

    Runs once per pipeline execution. Takes JD + config, produces SourcingPlan:

    • JD Analysis (Haiku): Extract must-haves, nice-to-haves, deal-breakers, seniority, location preferences
    • Company Research (Exa): Discover competitors, adjacent markets, funding announcements
    • GitHub Repo Discovery: Find popular repos in target tech stack (e.g., if JD is Python ML, discover popular ML repos)
    • Stream Configuration: Craft per-stream search queries:
    • GitHub: Search repos by language + topic tags + contributor filtering
    • Slack: Target LangChain, MLOps, Convergence communities
    • Company: List of target companies from replenisher (via ai_companies_master.csv)
    • Budget Allocation: Equal split by default (33% each for 3 streams)
    • Cost Estimation: Extrapolate to max_profiles, give Zack approval dialog if >$100

    Output: SourcingPlan dict consumed by Scout, Enricher, Evaluator

    Agent 2: Scout¶

    Runs 3 parallel streams (GitHub, Slack, Company), pushes LinkedIn URLs to url_queue.

    GitHub Stream: - For each repo in plan: query GitHub API for contributors - Extract real names, compute activity scores - Search LinkedIn for each name + "GitHub" + "engineering" - Push matches to url_queue

    Slack Stream: - For each Slack workspace (LangChain, MLOps, Convergence): read member list - Extract Slack profile → name + bio - Search LinkedIn for name + bio keywords - Push matches to url_queue

    Company Stream: - For each company in plan: use [[Company Raider]] to execute Apify batch search - Filter per post-filter rules - Push LinkedIn URLs to url_queue

    Dedup & Cooldown: - Skip sources scraped within 7 days (cooldown window) - Skip duplicates (normalize LinkedIn URLs before pushing) - Log scrape in pipeline_scrape_log table

    Sentinel Signal: - When all 3 streams complete, Scout pushes _SENTINEL to url_queue - Signals Enricher that discovery phase is done

    Agent 3: Enricher¶

    Drains url_queue, filters, enriches, post-filters, pushes to profile_queue.

    Pre-filter (Haiku): - For each URL batch: call Haiku with JD summary + LinkedIn URL - Haiku predicts "relevant" or "not relevant" - Drop non-relevant URLs before expensive Apify scrape (saves 30-40% Apify cost)

    Batch Scraping: - Accumulate URLs into batch of 200 (or 5 min timeout) - Call Apify batch scraper via actor supreme_coder~linkedin-profile-scraper - Parallel profile extraction: company, education, skills, experience - Cost: ~$0.003/profile (Apify cost gate: if batch >100, ask Zack before proceeding)

    Post-filter: - Run sourcing_filters.post_filter_profiles() on raw profiles - Check: years experience, location, company, seniority, title keywords - Drop failing profiles

    Output: - Passing profiles pushed to profile_queue - Stats updated: enriched count, Apify cost, pass/fail counts - Sentinel forwarded when url_queue exhausted

    Agent 4: Evaluator¶

    Drains profile_queue, PRISM-scores with Sonnet, generates outreach, saves to DB, posts to Discord.

    Batch Scoring (10 at a time): - Accumulate profiles from queue into batch - Call Sonnet with [[PRISM Scoring|PRISM evaluator]] prompt - Return score + reasoning for each

    Timing Signals: - Low score (1-2): "backlog" — file for future rounds - Medium score (3): "maybe" — needs follow-up research - High score (4-5): "pursue now" — generate outreach immediately

    Outreach Context: - Extract key differentiators (tech stack match, seniority fit, recent company) - Suggest opening angle (what to emphasize in message) - Flag potential concerns (gap in background, geographic concern)

    Database Save: - pipeline_results table: candidate, score, reasoning, timing, session_id - candidate_evals table: full PRISM report for later reference - sourced_profiles table: dedup against auto_sourcing engine

    Discord Post: - Summary embed per batch: count scored, distribution (5s, 4s, 3s, 2s, 1s) - Top candidates listed with score + opening angle - Post to channel (if configured) - DM Zack with final summary + cost breakdown

    Ashby Integration¶

    FORGE integrates with Ashby ATS via two methods:

    API Integration¶

    [[ashby.py|ashby.py]] uses Ashby REST API (no SDK) for: - List jobs (open, archived, all) - List candidates (applications) - Get interview schedules - Update candidate status - Archive candidates

    Auth: Basic auth (API key as username, blank password) All endpoints: POST to https://api.ashbyhq.com/{endpoint}

    CRITICAL: Ashby has NO public SDK and NO JavaScript export. The Ashby API is read-only for candidate listing — no bulk import capability. This drives the browser automation approach for pipeline scraping.

    Browser-Based Dashboard Scraping¶

    [[ashby_dashboard.py|ashby_dashboard.py]] uses [[forge-browser|forge-browser MCP]] to scrape Ashby UI:

    Sequence Templates Dashboard: - Navigate to https://app.ashbyhq.com/sourcing/sequence-templates - Extract: sequence name, enrolled count, reply rate, conversion rate, created_by, last_modified - Used for real-time pipeline visibility + outreach performance tracking

    Candidate Detail: - Open individual sequence - Extract applied candidates, interview stages, outcomes - Used for "which candidates are in which stage" queries

    Why browser automation? Ashby's API doesn't expose sequences or email campaign data. Scraping the UI is the only way to pull live pipeline metrics.

    Overnight Engine¶

    [[overnight_engine.py|overnight_engine.py]] runs autonomous deep work during VPS off-peak hours.

    Schedule: 4 cycles per night - 9:30 PM Pacific (5:30 UTC) - 12:30 AM (8:30 UTC) - 3:30 AM (11:30 UTC) - 6:30 AM (2:30 UTC)

    Cost Cap: $10/night (actual spend typically $3-6)

    Task Prioritization: 1. Continue yesterday's sourcing pipeline (enrich, score, post outreach) 2. Deep candidate research (GitHub activity, blog posts, open source) 3. Company intelligence (build profiles for target companies) 4. Stale outreach follow-ups (identify candidates who need nudging) 5. Knowledge base maintenance (re-index docs, prune duplicates)

    Per-Cycle Execution: - Pick highest-priority task - Run for up to 30 minutes - Log results + cost to overnight_state.json - Post summary to ⚡-forge channel

    Example overnight workflow: - 9:30 PM: Flush partial enrichment queue, score remaining profiles - 12:30 AM: Deep dive on top 10 candidates (GitHub activity, tweets, blog) - 3:30 AM: Replenish company list (mine previous candidates' employers, web discovery) - 6:30 AM: Post morning briefing with new candidates, overnight spend summary

    Auto-Sourcing Engine¶

    [[auto_sourcing.py|auto_sourcing.py]] runs autonomous sourcing 2x/day on VPS.

    Per-cycle pipeline: 1. Haiku generates sourcing niche (e.g., "Kafka engineers at Series B data startups") 2. Serper finds company domains in that niche 3. Apollo searches those domains for engineers 4. Dedup filter: skip URLs in sourced_profiles table 5. Apify scrapes only NEW profiles 6. PRISM scores 1-5 7. Post 4+ candidates to 🎯-talent channel, DM Zack summary

    Budget Cap: 2,000 Apify profiles/day. If hit → DM Zack for approval before proceeding.

    Configuration: - Title allowlist: software engineer, ML engineer, AI engineer, backend, data engineer, platform engineer - Seniority: senior, manager, entry - Title blocklist: founder, CEO, VP, director - JD: backend_engineer_ai_focused.txt

    Output: - Discord post to 🎯-talent (high-score candidates) - DM to Zack with summary (how many scored, cost spent, highlights) - Database: sourced_profiles table (LinkedIn URL → score → timestamp)

    ENG Sync Agenda Generation¶

    [[meeting_agenda.py|meeting_agenda.py]] auto-generates Friday 1:1 agenda for Zack + Hank meeting.

    Inputs: - Agenda template (templates/friday_meeting_agenda.md) - Open action items (meeting_action_items.json) — filtered to Zack/Hank/Unassigned - Recent FORGE changelog (tooling updates from past 7 days) - Previous meeting notes (analysis + unresolved items) - Ashby pipeline metrics (via ashby_dashboard: sequences, conversion rates, candidate stages)

    Output: - Pre-populated Friday agenda in Markdown - Action items grouped by owner - Overdue items flagged - Recent tooling updates listed - Candidate pipeline summary (stage breakdown, conversion funnel)

    Usage: "generate friday agenda" / "prep friday meeting" via Discord → generates agenda file, uploads to ⚡-forge channel

    Company Replenisher¶

    [[company_replenisher.py|company_replenisher.py]] auto-discovers new companies to raid.

    Strategy 1: Mine Existing Candidates - Query candidate_evals where score >= 3.0 - Extract previous employers from profile experience - Example: High-scorer from Cursor who previously worked at "Acme AI" → add "Acme AI" to raid list - Theory: If we scored them high, their previous companies are also worth raiding

    Strategy 2: Web Discovery - Haiku generates "trending AI startups + recent funding rounds" - Serper searches for companies matching keywords - Validate company exists + has engineering team - Add to ai_companies_master.csv

    Output: - ai_companies_master.csv (master list of target companies) - Feeds into Overnight Engine [[Company Raids]] task - Next cycle raids new companies discovered

    Sourcing Tracker¶

    [[sourcing_tracker.py|sourcing_tracker.py]] tracks sourced candidates and target companies via Discord.

    Candidate Tracking: - "s+ John Doe dba" → quick add (shorthand: s+ = source add) - Bulk add with role suffix parsing (dba, lead, support, relo) - Embed scoreboard with progress bars + streak tracking

    Company Pipeline: - Target count vs. sourced count - Conversion rate (sourced/target) - List active targets

    Features: - Quick add (shorthand syntax for speed) - Undo history (revert last batch add) - Display number → DB ID mapping (click "2" in list, get right record) - Export to text (HireIQ format for sync)
